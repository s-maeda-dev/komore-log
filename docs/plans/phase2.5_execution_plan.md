# 📋 Phase 2.5 実行計画書：バリデーションと入力チェックの実装

このドキュメントは、Phase 3 (AI連携) に進む前に、アプリケーションの堅牢性を高めるための入力チェック（バリデーション）を実装する手順をまとめたものです。

---

## 1. 概要
ユーザーからの入力内容が空であったり、文字数が多すぎたりする場合に、適切なエラーを返せるようにします。これにより、AIへの不要なリクエストを防ぎ、データの整合性を保ちます。

---

## 2. 詳細手順

### 2.1 依存関係の追加 (`pom.xml`)
バリデーション機能を使用するために必要なライブラリを追加します。

*   **追加するライブラリ**: `spring-boot-starter-validation`
*   **目的**: Bean Validation (Hibernate Validator) を有効にします。

### 2.2 DTO (Data Transfer Object) の導入
現在は `Map<String, String>` でリクエストを受け取っていますが、バリデーションを適用しやすくするために専用のクラスを作成します。

*   **クラス名**: `PostRequest.java`
*   **パッケージ**: `com.komore.komorelog.dto`
*   **実装内容**:
    *   `content` フィールド
    *   `@NotBlank`: 必須入力チェック（空文字禁止）
    *   `@Size(max = 300)`: 最大文字数チェック（300文字以内）

### 2.3 コントローラーの修正 (`PostController.java`)
作成した DTO を使用するように変更し、バリデーションを有効にします。

*   **変更点**:
    *   `createPost` メソッドの引数を `PostRequest` に変更。
    *   `@Valid` アノテーションを付与して、自動チェックを有効化。

### 2.4 例外ハンドラーの作成
バリデーションエラーが発生した際に、ブラウザへ分かりやすいエラーメッセージを返す仕組みを作ります。

*   **クラス名**: `GlobalExceptionHandler.java`
*   **パッケージ**: `com.komore.komorelog.exception`
*   **実装内容**:
    *   `@RestControllerAdvice` を使用。
    *   `MethodArgumentNotValidException` をキャッチして、エラーの詳細（どの項目がどうダメなのか）を返す。
